# EU AI Act → claude-governance Mapping Guide

**For**: Developers building high-risk AI systems for the EU market
**Purpose**: Map EU AI Act obligations (Articles 9-15) to `claude-governance` skills + agents you can invoke today, plus pointers to external runtime concerns
**Enforcement date**: 2 August 2026

> ⚠️ **NOT LEGAL ADVICE.** This guide is a developer reference, not legal counsel. Article references are based on Regulation (EU) 2024/1689 (Official Journal version of 13 June 2024) cross-verified with EC AI Act Service Desk. The Aug 2 2026 date is firm in current text but subject to Digital Omnibus harmonized standards conditionality. **Consult a qualified EU AI lawyer before using this guide for production compliance.** See `docs/research/eu-ai-act-obligations.md` for primary-source verified quotes per article.

> **Companion plugin**: For the full 7-step development workflow (research → requirements → design → breakdown → build-brief → review → deploy → monitor) that produces Annex IV-shaped evidence as a side effect, install [`pitimon/8-habit-ai-dev`](https://github.com/pitimon/8-habit-ai-dev) alongside this plugin. ## How to Use This Guide

The EU AI Act doesn't tell you _how_ to comply — it tells you _what_ to prove. This guide bridges the gap by mapping each obligation to a concrete `claude-governance` skill, agent, or template that produces audit-ready evidence — and points to external concerns (runtime monitoring, accuracy benchmarking) where this plugin doesn't own the work.

### 3-Step Workflow

1. **Scope check**: Run `/eu-ai-act-check --scope` first. If your AI system is **not high-risk** under Annex III or **not deployed in the EU**, you can skip this entire guide. Most internal tools, dev tooling, and non-safety AI fall outside scope.

2. **Map obligations to skills**: Use the table below to find which `claude-governance` skill or agent addresses each obligation. Some obligations route to **External** — runtime concerns (record-keeping logs, accuracy drift monitoring) that live in your application code or operational tooling, not this plugin.

3. **Generate evidence**: Each skill produces an artifact. Save it under `docs/compliance/eu-ai-act/<obligation>/` in **your project repository** with a date stamp. That folder becomes your audit trail.

### One-Time Bootstrap

Before your first compliance check, create the evidence folder structure in your project repo (the skills will NOT auto-create it — you control what gets versioned):

```bash
mkdir -p docs/compliance/eu-ai-act/{01-risk-mgmt,02-data-gov,03-tech-docs/annex-iv,04-records,05-transparency,06-oversight,07-accuracy,08-robustness,09-cybersecurity,reports,scope-decision}
echo "# EU AI Act Evidence — Generated by claude-governance compliance skills" > docs/compliance/eu-ai-act/README.md
git add docs/compliance/eu-ai-act/README.md && git commit -m "chore: bootstrap EU AI Act evidence folders"
```

This creates 9 obligation folders + reports + scope-decision directories.

### What This Guide Does NOT Do

- ❌ Make legal determinations — talk to a lawyer for Annex III classification
- ❌ Replace conformity assessment (Article 43) — that's a separate process
- ❌ Cover Articles 16+ (deployer obligations, post-market reporting) — provider-side only
- ❌ Cover Article 50 (general-purpose AI transparency) — different rule set
- ❌ Provide runtime telemetry (Articles 12 + 15 ¶1-3) — that's `pitimon/8-habit-ai-dev`'s `/monitor-setup` strategy plus your application logging

## Detailed Mapping

### 1. Risk Management System (Article 9)

**What the law wants**: Continuous, iterative risk identification and mitigation throughout the AI system lifecycle.

**How `claude-governance` helps**:

```
/governance-check security        # OWASP DSGAI 11-control checklist + secret scanning
/governance-check architecture    # service boundaries, irreversible-op safeguards
governance-reviewer agent         # deep multi-file review with severity grading
```

**Invocation example**:

```
/governance-check security
→ produces report with: known risks (DSGAI controls), mitigations, residual risk
→ save to: docs/compliance/eu-ai-act/01-risk-mgmt/2026-08-01-pr-123.md
```

**Why this works**: `/governance-check` already covers OWASP DSGAI (11 controls). Article 9 just requires you to keep the output as a dated artifact in your project repo.

### 3. Technical Documentation (Article 11)

**What the law wants**: Annex IV-compliant documentation showing compliance _before_ market placement.

**How `claude-governance` helps**:

```
/spec-driven-dev    # produces spec.md (Annex IV §1-3 — system + intended purpose)

```

**Invocation example**:

```

```

**Why this works**: `claude-governance`'s spec-driven workflow + ADR convention naturally produce most Annex IV §1-7 artifacts. Annex IV §6 (post-market monitoring plan) is external — see Obligation 4.

### 4. Record-Keeping (Article 12) — **External**

**What the law wants**: Automatic event logging during system operation for risk identification and post-market monitoring.

**This is outside `claude-governance`'s scope** — runtime telemetry lives in your application code and operational tooling. The plugin produces _strategy_, not _infrastructure_.

**How to satisfy**:

- **Strategy**: install `pitimon/8-habit-ai-dev`, run `/monitor-setup` to define logging granularity, retention, tamper-resistance
- **Implementation**: your application code (or a runtime telemetry system: Datadog, OpenTelemetry, ELK, etc.)
- **Evidence**: save the strategy doc to `docs/compliance/eu-ai-act/04-records/logging-config.md` in your project repo

> ℹ️ Article 12 does NOT specify retention duration, granularity, or immutability. Those are operational best practices but not legal requirements.

### 6. Human Oversight (Article 14) ⭐ Three Loops Integration

**What the law wants**: Effective human oversight — design enables shutdown, override, and prevents over-reliance.

**How `claude-governance` helps**:

```
governance-reviewer agent     # auto-triggered, Three Loops + Consequence classification
docs/adr/ADR-002              # 2D matrix (Task Type × Consequence) + irreversible-op override rule
```

**Invocation example**:

```
For any irreversible operation (production deploy, data deletion, credential rotation),
ADR-002 Consequence Override applies: classification becomes In-the-Loop regardless of task type.

Example design output for: "AI-assisted lending decision system"
→ Loan approval > $X → Irreversible consequence → In-the-Loop (human required)
→ Loan approval < $X → Contained consequence → On-the-Loop (AI proposes, human confirms)
→ UI rendering → Reversible consequence → Out-of-Loop (autonomous)
→ Save matrix to: docs/compliance/eu-ai-act/06-oversight/oversight-design.md
```

**🔗 Why this is the killer integration**: Article 14 mandates specific human-oversight capabilities (override, stop, intervene, disregard). The Three Loops design pattern documented in ADR-002 is one way to **satisfy** those capabilities — it is NOT a term used by EU law itself. When auditors ask "show me your human oversight design," you point to your design output and cite Article 14 ¶4(a-e) directly. The Three Loops table organizes _how_ you satisfy each ¶, but auditors will check your evidence against the regulation's text, not your framework labels.

> ⚠️ **Important framing**: The "Three Loops" model originates from human-autonomy teaming literature (Endsley 1999, DARPA), not EU law. Article 14 provides the legal _capabilities_ requirement; Three Loops + Consequence-Based Authorization is one valid design pattern that meets it.

### 7. Accuracy (Article 15 ¶1-3) — **External**

**What the law wants**: Appropriate accuracy levels with documented baselines and ongoing monitoring.

**This is outside `claude-governance`'s scope** — accuracy benchmarking and drift monitoring are runtime concerns.

**How to satisfy**:

- **Baseline capture**: install `pitimon/8-habit-ai-dev`, run `/review-ai` at model release time
- **Drift monitoring**: `/monitor-setup` defines the strategy; runtime ML observability (Arize, WhyLabs, MLflow, etc.) implements
- **Evidence**: save baselines + drift config to `docs/compliance/eu-ai-act/07-accuracy/baselines-v1.md`

### 9. Cybersecurity (Article 15 ¶5) ⭐ DSGAI Integration

**What the law wants**: Appropriate cybersecurity protecting model integrity, access, and supply chain.

**How `claude-governance` helps**:

```
/governance-check security              # OWASP DSGAI 11-control checklist
docs/compliance/DSGAI-MAPPING.md        # full mapping including bidirectional Art. 15 ¶5 cross-reference
hooks/secret-scanner.sh                 # blocks credential leakage at write-time (DSGAI02)
examples/ai-supply-chain-checklist.md   # DSGAI04 supply-chain hardening
```

**Invocation example**:

```
/governance-check security
→ output includes: 11 DSGAI controls pass/fail with evidence
→ cross-reference Art. 15 ¶5's 5 attack categories via DSGAI-MAPPING.md
→ save to: docs/compliance/eu-ai-act/09-cybersecurity/dsgai-audit.md
```

**🔗 Why this is the second killer integration**: Article 15 ¶5 names exactly 5 AI-specific attack categories (data poisoning, model poisoning, adversarial examples/evasion, confidentiality attacks, model flaws). OWASP DSGAI 11 controls provide the _concrete operational checklist_ for each category. Article 15 ¶5 = the "WHAT to defend against"; DSGAI = the "HOW to defend". See `docs/compliance/DSGAI-MAPPING.md` "EU AI Act Cross-References" section for the bidirectional table.

## What `claude-governance` Does NOT Cover

Be honest about scope:

| Out of Scope                          | Why                                                                            |
| ------------------------------------- | ------------------------------------------------------------------------------ |
| Annex III risk classification         | Legal judgment — talk to a lawyer                                              |
| Conformity assessment (Article 43)    | Separate certification process                                                 |
| Notified body interaction             | External certification body required                                           |
| Article 50 transparency (general AI)  | Different rule set                                                             |
| Articles 16-29 (deployer obligations) | This plugin is provider-side compliance enforcement, not deployment ops        |
| Runtime telemetry (Article 12)        | Strategy via `/monitor-setup` (8-habit); implementation via your runtime stack |
| Accuracy monitoring (Article 15 ¶1-3) | Strategy via `/review-ai` + `/monitor-setup` (8-habit); runtime ML obs tooling |
| 7-step development workflow           | Lives in `pitimon/8-habit-ai-dev` — install both for full coverage             |

For these, you need:

- **Legal counsel** for classification + certification
- **Runtime systems** (Datadog, OpenTelemetry, MLflow, Arize, etc.) for actual logging + drift detection
- **GRC tools** (OneTrust, Drata, etc.) for full compliance workflow management
- **`pitimon/8-habit-ai-dev`** plugin for workflow discipline that produces evidence as a side effect

`claude-governance` sits at the **compliance enforcement layer** — making sure the standards apply at write-time, commit-time, and PR-time. It's not the entire compliance stack, and that's by design.

## NIST AI RMF Cross-References

NIST AI RMF 1.0 functions overlap with EU AI Act Articles 9-15 substantially (~60-70% foundation per GLACIS community analysis at `https://www.glacis.io/guide-nist-ai-rmf-vs-eu-ai-act`). Note: NIST is voluntary US federal guidance; EU AI Act is regulatory. NIST coverage is a **starting point** for EU AI Act compliance, not a substitute. Specific EU-only gaps not addressed by NIST: prohibited AI practices (Art. 5), CE marking + conformity assessment (Art. 43, 48-49), EU AI database registration (Art. 71), 15-day serious incident reporting (Art. 73), explicit fines (€35M or 7% global revenue per Art. 99), authorized EU representative (Art. 22).

| NIST Function | EU AI Act Article(s) |
|---------------|----------------------|
| **GOVERN**    | Art. 16, 17, 26(6), 4 |
| **MAP**       | Art. 6, 9(1), Annex III, 10, 14, 13, 9(2), 27 |
| **MEASURE**   | Art. 15(1), Annex IV(2)(e-f), 9(6-7), 10(2)(f), 10(5), 43, 31-39 |
| **MANAGE**    | Art. 9(4), 9(2)(b), 20, 18, 12, 73, 72 |

See `docs/compliance/NIST-AI-RMF-MAPPING.md` for the canonical 3-way table (NIST function vs ISO 42001 clause vs EU AI Act article), the EU-only gap list with mitigation pointers, and the cited Microsoft-authored crosswalk PDF SHA-256 verification snippet.
